Below is a **full, end-to-end case workflow** you can implement for an insolvency case management app, including:

* **Stages + tasks per stage**
* **Validation gates** to move to the next stage
* **Automatic scheduling** based on the *notice date* (case creation date) and fallback **Company Settings**
* **Document recognition + auto-fill case details**
* **AI case-status summary agent**
* **Templates-Ro**: mail merge → email → deadline-safe scheduling
* **Creditor meeting** function in the sidebar
* Extra features that make it “best-in-class”
* A **Claude Opus prompt** to build + wire the whole app

## 1) Core rules you specified (made explicit)

### Case creation date

* **CaseCreationDate = NoticeDate extracted from the original notice document**
* If the notice doc has multiple dates, store them all, but designate one as **NoticeDate (source of truth)**.

### Deadlines and scheduling

When a case is created, tasks must be scheduled:

1. **From dates in the original notice**, if present (e.g., filing deadline, meeting date, objection period).
2. If not specified, from **Company Settings defaults** (e.g., “creditor claim deadline = NoticeDate + 30 days”, “send initial notice within 2 days”).
3. There must be a section to define **“deadline period calculated from case creation date”**:

   * Example: “Claim submission deadline = NoticeDate + X calendar days”
   * This can exist at multiple levels:

     * Global defaults
     * Company defaults
     * Case-type defaults
     * Case override

### Actions must be persisted as tasks

* Every action is a **Task** record with:

  * **mandatory deadline**
  * **assignee**
  * **default assignee = uploader** (unless the workflow says otherwise)
  * status, completion date, dependencies, escalation policy

### Key documents in the flow

* Upload → system should **recognise key document types**, extract:

  * parties
  * dates
  * obligations/actions
  * structured fields that populate the case

### Templates-Ro

* Templates in `Templates-Ro` must be:

  * **generated by deadline (never miss)**
  * **mail-merged**
  * **emailed** to relevant parties
  * parties must be attachable to **phase**, and attachable to **action/email**

### Creditor meeting

* A dedicated sidebar function: **“Call creditor meeting”**
* Must schedule tasks, generate notices, send emails, create calendar entries.

## 2) Data model (minimal but complete)

### Case

* `CaseId`, `CaseNumber`, `CaseType`, `Stage`, `Status`
* `NoticeDate` (source doc extracted; becomes CaseCreationDate)
* `CompanyId` (tenant boundary)
* `Court/Authority`, `Jurisdiction`, `InsolvencyPractitionerId`
* `KeyDeadlines` (structured: claim deadline, objection deadline, meeting date, report dates, etc.)
* `StageEnteredAt`, `StageCompletedAt`
* `AuditTrail` (or separate table)

### Party

* `PartyId`, `CaseId`, `Role` (Debtor, Creditor, Court, Administrator, Employee Rep, Bank, Tax Authority, Supplier, etc.)
* `Name`, `Identifiers` (VAT/CUI, reg no), `Address`, `Email`, `Phone`
* `PreferredDelivery` (email/print/both)
* `PhaseBindings` (or join table `PartyPhase`)

### Document

* `DocumentId`, `CaseId`, `UploadedBy`
* `DocType` (RecognisedType + Confidence)
* `StorageUrl`, `Hash`, `Version`
* Extracted:

  * `Summary`
  * `PartiesExtracted[]`
  * `DatesExtracted[]`
  * `ActionsExtracted[]` (each becomes a proposed task)
  * `FieldsExtracted` (JSON to map into case fields)
* `ReviewedBy`, `ReviewedAt`, `OverrideType`

### Task (Actions)

* `TaskId`, `CaseId`, `Stage`
* `Title`, `Description`, `Category` (Document, Email, Filing, Meeting, Call, Review, Payment, Report, Compliance)
* `Deadline` (mandatory), `DeadlineSource` (Notice / CompanyDefault / Manual)
* `AssigneeUserId` (mandatory), `CreatedByUserId`
* `Status` (Open/InProgress/Blocked/Done/Overdue/Cancelled)
* `Dependencies[]` (task ids)
* `EscalationPolicyId`, `ReminderScheduleId`
* `RelatedPartyIds[]`, `RelatedDocumentIds[]`, `RelatedEmailId`

### Template + Generated Letter

* `TemplateId`, `Name`, `Locale=ro`, `Stage`, `DocPurpose`
* `MergeSchema` (fields required)
* `GeneratedDocId`, `RenderedAt`, `SentAt`, `DeliveryStatus`

### Email

* `EmailId`, `CaseId`, `To/Cc/Bcc`, `Subject`, `Body`, `Attachments[]`
* `ScheduledSendAt`, `SentAt`, `Status`, `ProviderMessageId`
* Linked to tasks + parties + generated docs

### CalendarEvent

* `CalendarEventId`, `CaseId`, `Title`, `Start`, `End`, `Location`, `Participants`
* `ICSUrl` (optional), `SyncedExternal` (optional later)

### Company Settings (per tenant)

* Default deadline periods, e.g.:

  * `SendInitialNoticeWithinDays`
  * `ClaimDeadlineDaysFromNotice`
  * `ObjectionDeadlineDaysFromNotice`
  * `MeetingNoticeMinimumDays`
  * `ReportEveryNDays`
* Email settings, sending domain, signature blocks, templates mapping
* Working days rules (calendar vs business-day calculations)

## 3) Deadline engine (the “don’t miss deadlines” part)

### Deadline sources (priority)

1. **Notice-derived explicit date** (highest priority)
2. **Computed from NoticeDate + period** (Company default / Case-type default)
3. **Manual override** (requires reason + audit log)

### Computation rules

* Support:

  * calendar days vs business days
  * “next working day if weekend/holiday”
  * timezone fixed (Europe/Bucharest)

### “Never miss” enforcement

* For any task flagged `IsCriticalDeadline=true` (especially Template-Ro outputs):

  * Hard reminders: T-7, T-3, T-1, T-0 (configurable)
  * Auto-escalation:

    * assignee → team lead → partner/admin
  * System warning banners & daily digest emails
  * If deadline is within X hours and still blocked: trigger “**urgent queue**” and optionally auto-assign backup user

## 4) Stage-based workflow (tasks + gate validations)

You can implement stages as a **state machine**. Each stage has:

* required tasks (some auto-generated)
* required documents
* required data completeness checks
* allowed transitions

### Stage 0 — Intake & Case Creation

**Goal:** Create the case from the notice document and schedule the baseline tasks.

**Tasks (auto)**

* Upload original notice document (or confirm uploaded)
* Recognise notice doc type + extract NoticeDate (mandatory)
* Extract parties (debtor, authority/court, practitioner, known creditors if present)
* Generate baseline deadlines from notice or Company Settings
* Create initial task plan (per this workflow)
* Assign case owner / lead practitioner

**Key documents**

* Original Notice (the anchor document)

**Validation to progress**

* NoticeDate extracted and confirmed
* Case type chosen
* Debtor party exists with identifier (CUI/VAT/reg no)
* At least one responsible internal user assigned (Case Owner)
* Baseline deadlines computed and stored

---

### Stage 1 — Eligibility & Setup

**Goal:** Confirm basics, set scope, initialise statutory communications.

**Tasks**

* Verify debtor identity & registry details
* Confirm insolvency type (procedure type)
* Configure case-specific deadlines (override if needed)
* Create case-specific party list:

  * debtor representatives
  * court/authority
  * primary creditor list (even partial)
* Generate “Initial notices” from Templates-Ro (critical)
* Prepare filing checklist (if applicable)

**Key documents**

* Debtor registration extract / proof
* Appointment / authorisation documents (if required)

**Validation**

* Party roles complete: Debtor + Authority/Court + Practitioner
* Template-Ro initial notice generated (or scheduled) with valid merge fields
* At least one delivery method per key party (email or address)

---

### Stage 2 — Formal Notifications & Publication

**Goal:** All required parties notified, communications tracked, proof retained.

**Tasks (critical)**

* Mail merge and generate required notices (Templates-Ro)
* Email notices to parties (with attachments)
* Log delivery proof (email status + attachments)
* If physical delivery required: create print queue tasks
* Create/verify calendar dates (claim deadline, meeting dates if already known)

**Key documents**

* Proof of sending / delivery
* Published notice receipts (if applicable)

**Validation**

* All required notices sent to required recipients (tracked)
* Delivery statuses not in “failed” (or failure triaged with retry task)
* Claim submission window opened (dates stored)

---

### Stage 3 — Creditor Claims Collection & Register

**Goal:** Gather claims, validate, create the creditor register, prepare disputes.

**Tasks**

* Open claims intake (portal/email ingestion)
* For each claim received:

  * classify document type (claim form, evidence)
  * extract creditor identity, amount, priority, dates
  * create creditor party if new
  * create task: “Review claim” (assignee default = uploader/reviewer)
* Build the provisional creditor table/register
* Identify missing evidence → send request templates (Templates-Ro)
* Manage disputes/objections workflow

**Key documents**

* Creditor claims + supporting evidence
* Requests for missing info
* Objections/disputes

**Validation**

* Claim deadline reached OR “sufficient claims collected” rule satisfied (configurable)
* All received claims have a reviewed status (Accepted/Rejected/Needs Info)
* Creditor register generated and locked (versioned)

---

### Stage 4 — Asset & Liability Assessment

**Goal:** Understand estate, recoveries, contracts, litigation possibilities.

**Tasks**

* Collect debtor financials (balance sheet, bank statements, AR/AP, inventory)
* Identify assets and encumbrances (pledges, mortgages)
* Create asset list with valuations
* Identify avoidance actions / clawbacks (if applicable)
* Engage experts (valuation, legal) if needed (tasks + party roles)

**Key documents**

* Financial statements
* Asset valuation reports
* Contracts / litigation docs

**Validation**

* Asset register created with completeness threshold met
* Material liabilities reconciled with creditor register
* Recovery strategy documented (even as a short internal memo)

---

### Stage 5 — Creditor Meeting (sidebar function)

**Goal:** Convene meeting, publish agenda, capture resolutions, store minutes.

**Sidebar action: “Call creditor meeting”**
When triggered:

* User selects:

  * proposed meeting date/time/location (or online link)
  * agenda template
  * eligible attendee list (from creditor register)
* System generates:

  * calendar event
  * notices (Templates-Ro)
  * tasks: send notices, confirm quorum rules, prepare voting pack
* Automatically schedules:

  * notice send deadline = meeting date - minimum notice period (from Company Settings or law config)
  * reminders to internal team

**Tasks**

* Generate meeting notice pack (critical)
* Send meeting invites/notices
* Prepare voting register
* Record attendance and votes
* Upload minutes → extract resolutions → generate follow-up tasks

**Key documents**

* Meeting notice
* Agenda
* Voting register
* Minutes & resolutions

**Validation**

* Notices sent within minimum notice window
* Minutes uploaded and resolutions recorded
* Any required filings/notices arising from resolutions scheduled

---

### Stage 6 — Realisation & Distributions

**Goal:** Execute recovery plan, sell assets, distribute proceeds, keep compliance.

**Tasks**

* Create sale processes (auction/private sale)
* Approval workflow (internal + creditor/court if needed)
* Payment collection tasks
* Distribution schedule creation
* Generate distribution notices (Templates-Ro)
* Record payments, attach receipts

**Key documents**

* Sale agreements
* Proof of payment
* Distribution statements

**Validation**

* Distribution ledger balances
* Required approvals recorded
* Required notices sent

---

### Stage 7 — Reporting & Compliance (runs throughout, but stageable)

**Goal:** Periodic reports, statutory filings, audit-ready history.

**Tasks**

* Generate recurring reports (every N days from NoticeDate or stage-entry)
* File/report send tasks (Templates-Ro)
* Internal review + sign-off tasks
* Compliance checks (missing proofs, overdue tasks, incomplete registers)

**Key documents**

* Periodic reports
* Filing confirmations

**Validation**

* Reports generated + delivered on schedule
* No critical overdue compliance tasks (or exception approvals logged)

---

### Stage 8 — Closure

**Goal:** Close case, archive, final notices, retention.

**Tasks**

* Final account statement
* Final report + distribution summary
* Final notices (Templates-Ro)
* Archive all docs, export bundle
* Close open tasks or mark as cancelled with reasons
* Post-case QA checklist

**Key documents**

* Closure report
* Final notices
* Archive package

**Validation**

* All required closure documents generated and sent
* No open critical tasks
* Archive completed and immutable snapshot created

## 5) Document recognition & auto-fill (key doc types)

### Recognition pipeline

On upload:

1. Store file → hash/version
2. Classify doc type (ML + rules)
3. Extract:

   * parties
   * dates (and what each date “means”)
   * monetary amounts
   * obligations/actions
4. Produce:

   * **Document Summary**
   * **Structured Fields** mapped to Case/Party/Claim/Meeting/Distribution entities
5. Show a “Review extracted data” UI:

   * accept / edit / reject
   * confidence indicators
6. When accepted:

   * write changes into case entities
   * create tasks from extracted actions (mandatory deadline required; default from doc date rules)

### Key documents to highlight (typical)

* Original notice (anchors NoticeDate)
* Appointment/authorisation documents
* Creditor claims and evidence
* Creditor register versions
* Meeting notice / agenda / minutes
* Asset valuation reports
* Sale agreements
* Distribution statements
* Periodic reports
* Final report / closure notice

Every “key document” should:

* appear pinned in the stage timeline
* show extracted parties/dates/actions at a glance
* have a mandatory document summary field (auto-generated, editable)

## 6) AI case-status summary agent (what it should do)

### Inputs

* Case: current stage, deadlines, KPI counters (open tasks, overdue, critical upcoming)
* Tasks: grouped by stage + urgency
* Documents: each has stored summary + extracted actions + key fields
* Parties: key contacts, missing emails/addresses
* Events: next meeting dates, filings due

### Output (stored and displayed)

* “Current case status” narrative (5–10 bullet points)
* “What must happen next” (top 5 tasks)
* “Risks” (overdue, missing documents, failed deliveries)
* “Upcoming deadlines” (next 7/14/30 days)
* “Recent changes” (last uploads, last completed tasks)

### Storage

* Store each AI summary as:

  * `CaseSummaryId`, `CreatedAt`, `CreatedBy=system`, `Model`, `SnapshotJson`, `Text`
* Trigger summary refresh:

  * after document upload + extraction acceptance
  * task status changes
  * stage transitions
  * daily scheduled refresh (optional)

## 7) Templates-Ro: mail merge + email + tracking

### Template system requirements

* Templates have required merge fields (schema)
* Generation fails fast if required fields missing:

  * creates a blocking task: “Fix merge fields”
  * cannot progress stage if it’s a required send

### Mail merge flow (per template send)

1. Select template (or auto-selected by stage)
2. Resolve recipient parties (by role + phase bindings)
3. Render document (DOCX/PDF)
4. Create Email draft (subject/body from template metadata)
5. Attach generated docs
6. Schedule send time (immediate or at configured deadline)
7. Send via provider → record status → store proof

### Deadline guarantee

For “must never miss” templates:

* System schedules:

  * **Generation task** at T-(buffer)
  * **Approval task** optional
  * **Send task** at deadline
* If generation fails, escalation triggers immediately.
